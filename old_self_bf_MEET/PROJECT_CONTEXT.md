# PROJECT_CONTEXT.md (v1)
Context, glossary, and conventions for RL-based control with a stochastic latent process

## 0. Purpose

This file provides a shared vocabulary and minimal conventions so that:
- humans and code-generation agents (e.g., Codex) interpret terms consistently
- interfaces remain general without guessing hidden physics
- task cards can reference symbols unambiguously

This file is intentionally *non-physical-model-specific*.

---

## 1. Core objects and symbols

### 1.1 Episode and time steps

- **Episode**: one rollout of length T (variable), indexed by step i.
- **Step index**: i = 1, 2, ..., T
- **Max length cap**: n_max (a fixed integer upper bound)
- **Early termination**: episode may end before n_max when a termination condition triggers.

We assume *episodic RL* throughout.

### 1.2 Control action θ_i (high-dimensional, continuous)

- **Action** at step i: **θ_i**
- θ_i is a *continuous, high-dimensional* object.
  - It may be a vector, or a structured collection of continuous parameters.
  - We do not assume dimensionality or parameter meaning globally.

Notational convention:
- θ_≤i denotes {θ_1, θ_2, ..., θ_i} (control history)

### 1.3 Readout C_i

- **Readout** at step i: **C_i**
- C_i is the simulator’s measurable output (scalar or vector; unspecified globally)
- C_≤i denotes {C_1, C_2, ..., C_i} (readout history)

C_i is produced by the simulator conditioned on:
- latent process realization (unknown, stochastic)
- control action θ_i
- possibly history (θ_≤i, C_≤i) depending on simulator/measurement model

### 1.4 Latent / hidden process (general stochastic object)

- **Latent process**: an unknown stochastic object that influences readouts.
- It is NOT assumed to be a single scalar function f(t).
- It may be:
  - high-dimensional
  - structured (fields, vectors, multi-source noise)
  - time-dependent
  - partially observed

The latent process is generated by a **latent-process / signal generator**.

We use generic names:
- L : latent process (conceptual)
- ξ : randomness / noise seed(s) used to generate a latent realization
- τ : a latent trajectory/realization (one sampled instance of L for an episode)

Important:
- By default the agent/policy does NOT directly observe L or τ unless a task explicitly permits it.
- L may be used for debugging/validation logging only.

## 2. Observations (o_i) and history dependence

### 2.1 Observation o_i

- **Observation** at step i: **o_i**
- o_i is what the RL agent receives from the environment wrapper.

Observation is intentionally flexible and may include:
- current readout C_i
- readout history C_≤i
- control history θ_≤i (likely needed)
- time features (i, normalized time)
- auxiliary diagnostics (if allowed)

We do NOT assume Markov state.
The project is compatible with POMDP settings.

### 2.2 History conventions

- θ-history: θ_≤i
- C-history: C_≤i
- Full observed history (conceptual): H_i = (θ_≤i, C_≤i, aux_≤i)

Observation builder is responsible for mapping histories into o_i.

## 3. Reward functional

### 3.1 Reward styles

Reward is unspecified and must remain configurable:

- **Step-wise reward**: r_i computed each step
- **Terminal reward**: R computed at episode end
- **Combined**: both step-wise and terminal

Conventions:
- r_i: per-step reward at step i
- R: terminal reward (could be a function of entire trajectory)
- Return may be sum of r_i, or R, or another aggregation (unspecified).

### 3.2 Reward inputs (conceptual)

Reward functions may depend on any of:
- θ_≤T, C_≤T
- termination reason
- auxiliary info
- Monte Carlo aggregates (if multiple latent samples used)

Reward computation is separated from simulator and environment stepping logic.

---

## 4. Termination semantics

### 4.1 Termination flags

At each step i:
- done ∈ {True, False}
- reason: string label describing why termination happened

Two termination mechanisms must be supported:
1) **Hard cap**: i reaches n_max
2) **Early termination**: threshold/condition triggers (user-defined)

Examples of early termination triggers (non-binding):
- readout exceeds/under-runs threshold
- reward reaches target
- constraint violation (e.g., action bounds)
These are only examples; not assumed globally.

---

## 5. Monte Carlo (MC) simulation semantics

Latent process is stochastic; readouts are stochastic.

We keep MC semantics open. The system must support:
- single-sample per episode (one latent realization)
- multi-sample per episode (K latent realizations) with aggregation
- batch-level MC (parallel episodes)

Notation:
- K: number of MC samples for a given aggregation (if used)
- C_i^(k): readout at step i under latent sample k
- Aggregated readout: \bar{C}_i (optional concept)
- Aggregated reward: \bar{r}_i or \bar{R} (optional concept)

The aggregation rule (mean, median, risk-sensitive, etc.) is configurable.

---

## 6. Interfaces: conceptual responsibility map

- Latent generator:
  - samples latent realizations/trajectories
  - controls randomness and reproducibility

- Simulator:
  - maps (latent, θ_i, history?) → readout C_i (+ aux)

- Observation builder:
  - maps histories (θ_≤i, C_≤i, aux_≤i) → o_i

- Reward functional:
  - maps trajectory record → r_i / R (configurable style)

- Termination:
  - maps current step and history → (done, reason)

- Environment wrapper:
  - orchestrates the above; exposes RL-facing `reset()` and `step()` semantics

---

## 7. Non-goals / Avoided assumptions

This project context intentionally does NOT assume:
- a specific physical Hamiltonian / master equation form
- whether θ_i corresponds to pulses, gates, amplitudes, phases, etc.
- whether C_i is scalar or vector
- a specific RL algorithm (PPO/DDPG/SAC/etc.)
- a specific observation encoding (concat vs RNN vs transformer)
- a specific configuration format (yaml/dict/dataclass)

All of these must be introduced only in task-level specs.
